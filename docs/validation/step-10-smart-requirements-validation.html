Detailed FR Analysis — Requirements Needing Improvement

FR13 — Storm Project Archival

"Super Admin can archive a closed storm project, preserving all associated data for a minimum of one year."

Measurability: 2/5 — "Preserving all associated data" is not testable as written. What constitutes "all associated data"? Does it include audit logs, uploaded photos at original resolution, draft assessments, or only approved final records? The one-year retention target is good, but the scope of what is preserved is ambiguous.

Specificity: 3/5 — The archival action is clear (Super Admin triggers it on a closed project), but the behavior of archived projects is undefined. Can archived projects be viewed? Searched? Exported? Is the archive reversible?

Suggested improvement: "Super Admin can archive a closed storm project. Archived projects retain all tickets, assessments (including photos at upload resolution), contractor records, and audit logs. Archived data remains read-only and queryable (searchable via project dashboard). Data is retained for a minimum of one year after project close-out date. Archival is reversible by Super Admin (project can be unarchived to 'Closed' status)."

FR22 — Assessment Form Structured Data

"The assessment form captures structured data (damage type, component affected, severity, span count, measurements, notes)."

Measurability: 2/5 — The parenthetical list reads as illustrative examples, not a definitive field specification. An implementer cannot determine from this requirement alone whether the list is exhaustive or representative. The PRD's "Deliberately Cut" section states the form is "one fixed assessment form, hard-coded fields," but FR22 does not enumerate those hard-coded fields.

Specificity: 3/5 — "Measurements" is vague (of what? in what unit?). "Notes" is unbounded. "Component affected" does not enumerate the component taxonomy (pole, primary, secondary, service, weatherhead) that is well-defined elsewhere in the PRD (product brief section).

Suggested improvement: "The assessment form captures the following fixed fields per ticket: (1) Component affected (enum: Pole, Primary, Secondary, Service/Neutral, Weatherhead/Meter), (2) Damage present per component (Yes/No), (3) Damage type (enum per component, e.g., Broken, Leaning, Damaged for Pole), (4) Severity (enum: Minor, Moderate, Major, Critical), (5) Span count (integer, for wire components), (6) Free-text notes (optional, max 1000 characters). Exact enum values will be defined in the data dictionary prior to implementation."

FR24 — Photo GPS and Timestamp Capture

"Photos automatically capture GPS coordinates and timestamp."

Measurability: 2/5 — "Automatically" is untestable without defining the mechanism. Are GPS coordinates extracted from EXIF metadata of the captured photo? Or does the app independently request browser geolocation at the moment of photo attachment? What happens if the device has location services disabled? What precision is required?

Attainability: 3/5 — On the web platform (Safari PWA on iPad), extracting EXIF GPS from photos taken via <input type="file" capture="environment"> is unreliable — EXIF stripping varies by browser and OS version. Independently requesting geolocation via the Geolocation API requires user permission and may fail in low-signal field environments. The requirement as written assumes reliability that may not exist.

Suggested improvement: "When a photo is attached to an assessment, the system records GPS coordinates and timestamp. Primary method: browser Geolocation API captures device position at the time of photo upload. Fallback: EXIF metadata extraction from the image file. If neither source is available (user denied location permission or no GPS signal), the photo is accepted but flagged as 'No Location Data' for reviewer attention. Required GPS precision: within 50 meters."

FR32 — In-App Notification for Assessment Status

"Contractors receive in-app notification that their assessment was approved or rejected."

Measurability: 2/5 — "In-app notification" is undefined as an implementation concept. Is this a toast/banner that appears on next login? A notification badge? A dedicated notifications list/inbox? A real-time push while the app is open? The test criteria for "contractor was notified" depend entirely on which mechanism is chosen.

Attainability: 3/5 — The MVP explicitly cuts SMS notifications and push notifications. Without these, the only notification surface is something the contractor sees when they open the app and navigate to the right screen. If the contractor does not open the app, they receive no notification at all. This may undermine the stated purpose of the requirement.

Suggested improvement: "When a Super Admin approves or rejects an assessment, the system updates the ticket status visible on the contractor's dashboard (FR36). A visual indicator (badge with unread count) on the contractor's ticket list highlights assessments with status changes since the contractor's last view. The contractor sees the approval or rejection with review notes (if rejected) when they open the ticket detail. No push or SMS notification in MVP — notification is passive (visible on next app open)."

FR36 — Contractor Personal Dashboard

"Contractors see a personal dashboard showing their assigned tickets and submission statuses."

Measurability: 3/5 — The dashboard content is only partially specified: "assigned tickets and submission statuses." What status groupings should be displayed? Should it show counts? A list? Both? Should rejected assessments be visually differentiated from pending ones? Without a minimum content specification, the acceptance test is subjective.

Suggested improvement: "Contractors see a personal dashboard showing: (1) A count of tickets grouped by status (Assigned, In Progress, Submitted, Approved, Rejected), (2) A scrollable list of all assigned tickets with ticket address, priority, and current status. Rejected assessments are visually highlighted with a 'Needs Revision' indicator."

Detailed NFR Analysis — Requirements Needing Improvement

NFR13 — Rapid User Scaling

"System accommodates rapid user growth from 0 to 200+ contractors during storm activation within hours."

Specificity: 3/5 — "Rapid" and "within hours" are subjective. Does this mean 200 accounts can be created and logged in simultaneously within 2 hours? Or that the system does not degrade when 200 accounts exist?

Measurability: 2/5 — No concrete test criteria. What does "accommodate" mean — that account creation works? That concurrent login works? That response time stays within NFR5 targets? The requirement is essentially a restatement of the scaling scenario without a testable threshold.

Attainability: 3/5 — On Supabase free/pro tier, 200 concurrent connections may require specific configuration. This should be validated against the Supabase plan selected.

Suggested improvement: "The system supports creation of up to 200 contractor accounts and 50 concurrent authenticated sessions without performance degradation (NFR5 response time targets maintained). Account creation (FR1) functions correctly when performed in batch (e.g., 20 accounts created within a 10-minute window). Validated against Supabase Pro tier connection limits."

NFR16 — Uptime Target

"System targets 99.5% uptime during active storm projects (Supabase-managed infrastructure)."

Measurability: 2/5 — "Targets" is aspirational language, not a commitment. 99.5% uptime is approximately 3.65 hours of allowed downtime per month — but there is no mechanism defined to measure, monitor, or alert on uptime. There is no defined measurement window (per storm? per month?).

Attainability: 3/5 — Supabase's own SLA on the Pro plan is 99.9% for database, but the overall system uptime depends on the application layer (Vercel/Netlify hosting), Supabase Auth, and Supabase Storage — all of which are outside the solo developer's control. The PRD cannot guarantee what depends on third-party SLAs.

Suggested improvement: "System availability targets 99.5% during active storm projects, as measured by successful response to a health-check endpoint over the storm project's duration. This target is dependent on Supabase managed infrastructure SLAs (database, auth, storage) and the hosting provider's SLA. The application layer (client code) shall not introduce single points of failure beyond infrastructure dependencies. Uptime monitoring shall be configured before first storm deployment."

NFR18 — Draft Data Preservation on Interruption

"If a browser tab or session is interrupted mid-form, draft data (FR26) is preserved locally and recoverable."

Measurability: 2/5 — "Preserved locally" is ambiguous. In localStorage? sessionStorage? IndexedDB? Each has different persistence characteristics (sessionStorage is lost on tab close, localStorage persists but has size limits, IndexedDB persists and scales but requires Dexie.js which is listed as Phase 2 offline infrastructure).

Attainability: 2/5 — This is the most problematic NFR. The MVP is explicitly "online-only" with offline mode (Dexie.js, service workers) deliberately cut to Phase 2. Yet this requirement implicitly requires local data persistence — which is the foundation of offline capability. Using localStorage for draft persistence is feasible but contradicts the architectural decision to defer local storage infrastructure. The requirement needs to either (a) be explicit about using a lightweight mechanism (localStorage/sessionStorage) distinct from the Phase 2 offline architecture, or (b) be acknowledged as accepting data loss on interruption in MVP.

Specificity: 3/5 — "Interrupted" could mean tab closed, browser crashed, device lost power, accidental navigation, or session timeout (NFR11). Each scenario has different recoverability characteristics.

Suggested improvement: "Draft assessment data (FR26) is persisted to the server via auto-save on each form section completion. If a session is interrupted (tab closed, navigation away, session timeout), the contractor can resume the draft from the last auto-saved state on next login. Auto-save triggers on: (a) section/tab completion, (b) every 60 seconds of active editing. This is distinct from Phase 2 offline storage — auto-save requires an active network connection. If the network is unavailable at auto-save time, the system displays a warning but does not block continued editing."

Cross-Cutting Observations

1. Measurability is the Weakest Dimension Across Both FRs and NFRs

The most common gap is missing acceptance test criteria. Requirements describe what the system does but do not always define how you would verify it works correctly. This is a common PRD pattern when the author is domain-expert (knows what "right" looks like) but the implementer may not share that intuition. Adding a brief "Acceptance Criteria" annotation to flagged FRs would close these gaps.

2. NFR18 Creates an Architectural Tension with "Online-Only" MVP

The requirement for local draft preservation (NFR18) implicitly introduces local storage into an MVP that explicitly defers offline/local storage to Phase 2. This is the single most important clarification needed before implementation. The suggested resolution (server-side auto-save) aligns with the online-only constraint while preserving the user intent of the requirement.

3. The Assessment Form (FR22) Needs a Companion Data Dictionary

FR22's field list is illustrative, not exhaustive. The product brief's "Assessment Data Complexity" table (pole, wire, equipment, safety, service, restoration, photos) provides much richer detail. The FR should either enumerate exact fields or explicitly reference the data dictionary as a companion artifact. This is a common gap when a PRD is derived from a product brief — the brief's detail gets summarized too aggressively.

4. Strong Alignment Between Scope Cuts and Requirement Quality

The PRD's "Deliberately Cut from MVP" table is excellent and directly improves the SMART quality of the remaining requirements. By cutting offline mode, configurable forms, SMS, duplicate detection, and photo optimization, the remaining 36 FRs are tightly scoped and realistically attainable within 30 days. This discipline is reflected in the high Attainability scores (4.83 avg for FRs).

5. Admin Role (FR5) May Need Refinement

FR5 states Admin cannot create storm projects, enter tickets, or assign contractors. However, the MVP scope table says "Marcus operates with same capabilities as Jeanie in MVP (no separate admin layer)." This is a potential contradiction — though FR5 scores adequately on SMART dimensions (avg 4.4), the implementation team should confirm whether Admin restrictions are enforced in MVP or deferred. The FRs as written (FR5 restricting Admin) are the more conservative and safer interpretation.
