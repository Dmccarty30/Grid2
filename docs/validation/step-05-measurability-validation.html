A.4 -- Top 5 FRs Needing Improvement

FR22 (Line 476) FORMAT + IMPLEMENTATION DETAILS

Current: "The assessment form captures structured data (damage type, component affected, severity, span count, measurements, notes)."

Issues: (1) No actor -- describes form content, not user capability. (2) Enumerates specific data fields, which are implementation details that will change and are better placed in a data dictionary or form specification.

Suggested rewrite: "Contractors can enter structured damage data for each assessment, including damage type, affected component, severity classification, and supporting measurements." -- Move the exhaustive field list to a separate form specification document.
FR4 (Line 446) FORMAT

Current: "The system enforces role-based access -- Contractors cannot access admin functions."

Issues: Uses "the system" as actor. Describes a constraint rather than a user capability. Mixes a system behavior with an access rule.

Suggested rewrite: "Contractors can access only contractor-scoped functions; admin-only functions are not visible or accessible to the Contractor role." Alternatively, move to a security/access control specification.
FR20 (Line 471) FORMAT

Current: "Tickets have a visible status (Unassigned, Assigned, In Progress, Assessment Submitted, Under Review, Approved, Rejected)."

Issues: Describes a data attribute of tickets, not a user capability. No actor. The enumerated status values are implementation details better placed in a data model or enum specification.

Suggested rewrite: "All users with ticket access can view the current status of each ticket, reflecting its lifecycle stage from creation through approval." Move the specific status enum to the data model.
FR24 (Line 479) FORMAT

Current: "Photos automatically capture GPS coordinates and timestamp."

Issues: No actor. Describes automatic system behavior. Could arguably be an NFR (data quality attribute) rather than an FR.

Suggested rewrite: "When a contractor uploads a photo, the system records the GPS coordinates and timestamp associated with the capture." This preserves the intent while clarifying the triggering actor and the system behavior.
FR32 (Line 489) FORMAT

Current: "Contractors receive in-app notification that their assessment was approved or rejected."

Issues: "Receive" is passive -- it describes something that happens to the user, not a capability they exercise. The notification is a system-initiated action.

Suggested rewrite: "Contractors can see an in-app notification when their assessment is approved or rejected." This frames the notification as something accessible/viewable by the actor.

B.3 -- Top 5 NFRs Needing Improvement

NFR14 (Line 533) FAIL -- NO METRIC

Current: "Database and storage architecture supports multiple concurrent storm projects across different utility clients."

Issues: "Multiple concurrent" is a vague quantifier. No numeric target. No performance threshold. Not testable as written.

Suggested rewrite: "The system supports at least 5 concurrent active storm projects, each with up to 2,000 tickets and 50 assigned contractors, with no degradation below NFR1--NFR5 performance targets."
NFR15 (Line 534) FAIL -- NO METRIC

Current: "Photo storage scales elastically -- storm projects may generate thousands of images per event."

Issues: "Scales elastically" is aspirational, not measurable. "Thousands" is a vague quantifier. No storage capacity target, no upload throughput target.

Suggested rewrite: "The system supports storage of at least 50,000 photos per storm project (at up to 800KB each, approximately 40GB per project) without requiring manual infrastructure scaling or impacting upload performance defined in NFR2."
NFR19 (Line 541) FAIL -- SUBJECTIVE + MISPLACED

Current: "System provides clear error messages when submission fails, with guidance to retry."

Issues: "Clear" is a subjective adjective. "Guidance to retry" is undefined. This reads more like a functional UX requirement than a non-functional quality attribute. Not testable without defining what "clear" means.

Suggested rewrite: Move to FRs as: "When a form submission fails, the system displays an error message identifying the failure reason and provides a one-tap retry action." If keeping as NFR, define: "Error messages are written at a 6th-grade reading level, include the specific failure reason, and include an actionable next step. Validated via UX review checklist."
NFR9 (Line 525) WARN -- IMPLEMENTATION DETAIL + BORDERLINE FUNCTIONAL

Current: "Row-Level Security (RLS) enforced at the database level -- contractors can only access their own data and assigned tickets."

Issues: Names a specific implementation mechanism (Supabase RLS). The data isolation requirement is functional (access control). An NFR should state the security quality attribute, not prescribe the implementation.

Suggested rewrite: NFR: "Data isolation between contractors is enforced such that no contractor can read, write, or enumerate another contractor's personal data or unassigned tickets, verified by automated penetration testing." Move "RLS at database level" to an architecture decision record.
NFR4 (Line 517) WARN -- UNDEFINED THRESHOLD

Current: "System supports up to 50 concurrent users without performance degradation."

Issues: "Without performance degradation" is undefined. Does it mean all NFR1--NFR5 targets still hold? Or some relaxed threshold? Without linking to specific performance baselines, this is not verifiable.

Suggested rewrite: "With 50 concurrent authenticated users performing mixed operations (form submissions, dashboard views, photo uploads), all performance targets in NFR1--NFR5 remain within their stated thresholds, as verified by load testing."

Rationale: The PRD's functional requirements are strong -- testable, specific, and free of subjective language. This alone prevents a CRITICAL rating. However, the non-functional requirements have measurability gaps (particularly in scalability and some security NFRs) that could lead to ambiguity during implementation and acceptance testing. Recommended Actions (Priority Order)

  1. Add numeric bounds to NFR14 and NFR15 -- These scalability NFRs are currently untestable. Define concrete numbers (project count, photo count, storage capacity).
  2. Rewrite NFR19 -- Remove subjective "clear", define error message requirements with testable criteria, or move to FRs.
  3. Link NFR4 to baseline performance targets -- Define "without performance degradation" by referencing NFR1--NFR5 explicitly.
  4. Separate implementation from quality in NFR9 -- State the data isolation requirement; move RLS to architecture documentation.
  5. Add network/load context to NFR3 and NFR5 -- NFR1 and NFR2 set a good example by specifying "4G/LTE connection"; apply the same rigor to NFR3 and NFR5.
  6. Consider reclassifying NFR8, NFR9, NFR10 -- These describe access control behavior. Consider moving the functional component to the FR section and keeping only the security quality attribute (e.g., "no unauthorized access detected in penetration testing") as the NFR.
  7. Standardize FR format -- Rewrite the 15 FRs that use "the system" or describe constraints to follow "[Actor] can [capability]" format for consistency.
