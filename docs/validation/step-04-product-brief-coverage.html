Summary of MVP Scope Misalignment

The two documents describe entirely different MVP philosophies:

  • Brief's Phase 1 = "Foundation" -- auth, roles, onboarding, dashboard shell, permission groups, PWA scaffold. NO tickets, NO assessments, NO storm operations. The operational features begin in Phase 2.
  • PRD's Phase 1 = "One Storm" -- auth, onboarding, storm project CRUD, ticket management, damage assessment, review workflow, dashboard with live data. This is a fully operational end-to-end storm workflow, online-only, with hard-coded forms.

These are not minor differences. The PRD has essentially redefined what "MVP" means -- from a foundation-first approach to an operations-first approach. Both are valid strategies, but the documents are not aligned.

Significant Divergences

DIVERGENCE 1 (CRITICAL): Authentication Architecture

Brief: "Authentication via Microsoft Entra ID (Azure AD) for SSO with existing M365 tenant, with Supabase for data layer" -- this is specified as a Phase 1 core feature.

PRD: "Supabase Auth (email/password)" -- no mention of Entra ID anywhere in the PRD. Auth reliability metric explicitly references "Supabase Auth (email/password)."

Impact: This is a fundamental architectural choice. Entra ID provides SSO with the company's existing Microsoft 365 tenant -- employees would log in with the same credentials they use for email. Supabase email/password creates a separate credential silo. The PRD's approach is simpler to implement (important for a solo non-developer on a 30-day timeline) but abandons the Brief's SSO vision entirely. The existing codebase (grid-electric-docs) confirms Supabase Auth with email/password is the implemented approach.

Resolution needed: Confirm whether Entra ID SSO is a hard requirement or if Supabase email/password is the accepted direction. If Entra ID is desired long-term, document it as a Phase 2 migration path.

DIVERGENCE 2 (CRITICAL): MVP Scope Philosophy

Brief Phase 1: "Foundation" -- authentication, roles, permission groups, onboarding, dashboard shell, PWA scaffold. Explicitly states "Damage assessment form" and "Ticket management" are Phase 2, OUT of scope for MVP.

PRD Phase 1: "One Storm" -- a fully operational end-to-end storm workflow including ticket management, damage assessment, and review. Explicitly cuts permission groups and offline scaffold.

Impact: These are opposite MVP strategies. The Brief builds infrastructure first (auth + onboarding + shell) and adds operations second. The PRD delivers a complete operational loop first (the whole storm workflow) and refines infrastructure second. Both are valid, but stakeholders referencing the Brief will have different expectations than developers referencing the PRD.

Resolution needed: Decide which MVP strategy is authoritative. The PRD's "One Storm" approach is arguably more pragmatic for proving value in 30 days, but the Brief's "Foundation" approach is more architecturally sound for long-term extensibility.

DIVERGENCE 3 (MODERATE): Permission Groups

Brief: Permission groups (Billing, Tickets, Users, Inventory, Reports) are a Phase 1 core feature (#3 of 10). The Brief emphasizes "features hidden, not disabled" as a key UX principle and lists it as a differentiator.

PRD: No permission groups in Phase 1. "Marcus operates with same capabilities as Jeanie in MVP." Permission groups are not explicitly scheduled in any phase.

Impact: The Brief considers this a foundational differentiator. Its absence from the PRD entirely (not even tracked in Future Functional Requirements) is a gap.

DIVERGENCE 4 (MODERATE): Offline/PWA Foundation

Brief: Phase 1 includes "PWA Shell -- Installable, offline-ready scaffold -- Dexie.js, service worker, background sync foundation."

PRD: Phase 1 is explicitly "Online-only. No service workers, no offline. Add in Phase 2 after core works."

Impact: The Brief envisions laying the offline groundwork in Phase 1 even if full offline capability is Phase 2. The PRD defers all offline infrastructure. This means Phase 2 will require retrofitting offline support rather than activating a pre-built scaffold.

DIVERGENCE 5 (LOW): Onboarding Scope

Brief: Full onboarding wizard with PPE acknowledgment, training modules, safety video completion, code of conduct sign-off.

PRD: "Simple Onboarding -- Name, phone, email, W-9 upload, insurance upload. One fixed form." No PPE, training, or acknowledgments in MVP.

Impact: The PRD deliberately strips onboarding to minimum viable. This is a reasonable scope cut for 30 days, but the Brief's Phase 1 envisions a richer wizard. The PRD should acknowledge this as a known simplification with a Phase 2 expansion path.

Recommendations

  1. Resolve the authentication divergence (D1). Confirm with the product owner whether Microsoft Entra ID SSO is required or if Supabase email/password is the accepted direction. If Entra ID is desired, add it to a specific phase in the PRD. If Supabase Auth is confirmed, update the Product Brief to reflect the change.
  2. Reconcile the MVP scope (D2). The PRD's "One Storm" approach is more pragmatic for proving value quickly. If this is the accepted direction, the Product Brief should be updated to match. Alternatively, designate the PRD as the authoritative implementation plan and the Brief as the strategic vision document, with a note explaining the divergence rationale.
  3. Track permission groups (D3/G1). Add permission groups to the PRD's Future Functional Requirements with a target phase. This is identified as a differentiator in the Brief and should not disappear from the roadmap.
  4. Formalize North Star metrics (G2/G3). Add "Ticket First-Attempt Acceptance Rate > 95%" and "Project Close-Out Time 40% reduction" to PRD Success Criteria.
  5. Add Baseline Capture requirement (G4). Include the pre-launch baseline measurement mandate from the Brief. Without baselines, success metrics cannot be evaluated.
